# Zen Canvas — Codex 工作约束（UI/UX v2）

本仓库正在进行一次完整的 UI/UX 重构。任何前端改动都必须遵守本文件与 `docs/design/UI_UX_REDESIGN_MASTER_SPEC.md`。

## 产品目标

Zen Canvas 是一个本地优先的个人文件生命周期管理助手。界面应像成熟的消费级桌面应用，而不是工程控制台、管理后台或组件展示页。

目标不是“增加更多玻璃、圆角和动画”，而是：

- 更少一级入口；
- 更少同时可见的信息；
- 每个页面只有一个明确主动作；
- 技术参数渐进式披露；
- 所有危险操作先预览、可撤销、可恢复；
- macOS 与 Windows 使用各自符合平台习惯的窗口外壳。

## 不可违反的设计规则

1. **一个页面只能有一个主要 CTA。** 同一作用的按钮不得在页面多个位置重复出现。
2. **禁止以卡片堆叠代替信息架构。** 能用留白、分组、分隔线解决时，不新增完整圆角卡片。
3. **禁止把内部实现直接暴露给普通用户。** Batch Size、并发、Token、原始 JSON、数据库状态、模型调试信息只能进入高级/开发者设置。
4. **一级导航固定为：概览、文件库、整理建议、历史记录。** 自动化规则与设置属于次级入口。
5. **扫描不是长期一级页面。** 扫描通过概览页、空状态或命令面板触发。
6. **整理建议与预览执行必须形成一个连续流程。** 不得重新拆成两个相互跳转的一级页面。
7. **恢复记录、清理记录和整理记录统一进入历史记录。**
8. **颜色只表达稳定语义。** 蓝=主要/信息，绿=成功/安全，黄=需确认，红=危险/失败，灰=中性。
9. **不要依赖颜色单独表达状态。** 同时使用文字、图标或形状。
10. **不得使用浏览器原生 `confirm`、`alert`。** 使用统一的应用内 Dialog/Sheet。
11. **默认界面不显示大段安全说明。** 安全应通过“预览、影响摘要、可撤销、永久删除数量”等机制体现。
12. **禁止无意义动效。** 不对整页卡片做批量弹跳或缩放。优先淡入、内容切换和进度反馈，并尊重 `prefers-reduced-motion`。
13. **所有文件列表都必须考虑键盘操作、多选、右键菜单、双击打开和平台差异。**
14. **文案必须面向普通用户。** 避免“调度、生命周期、候选分层、规则权重”等内部术语直接成为主界面标签。
15. **不允许为了通过旧 UI 测试而保留错误结构。** 旧测试与新产品结构冲突时，应更新测试并说明原因。

## 代码与组件要求

- 页面只负责组合，不在单个 View 中继续堆积数百行 UI 状态。
- 提取并复用：`AppSidebar`、`GlobalSearchTrigger`、`InspectorPane`、`FileList`、`SelectionBar`、`ProgressFlow`、`ConfirmSheet`、`EmptyState`、`InlineNotice`。
- 设计 token 集中管理，禁止在页面中大量散落任意透明度、阴影和圆角值。
- 新增组件必须覆盖：默认、hover、active、focus-visible、disabled、loading、error 状态。
- Windows 和 macOS 的标题栏、窗口按钮与系统文案应按平台适配。
- 每一阶段完成后必须运行：
  - `npm run typecheck`
  - `npm test`
  - 相关新增 UI 测试
  - `npm run build`

## Codex 执行方式

每次只完成一个阶段或一个页面，不允许一次性重写全部前端。

每次任务结束必须输出：

1. 修改了哪些文件；
2. 删除了哪些冗余信息或重复操作；
3. 当前页面唯一主动作是什么；
4. 已覆盖哪些状态；
5. 运行了哪些验证；
6. 仍有哪些未解决的设计债务。

当实现与总规范冲突时，以 `docs/design/UI_UX_REDESIGN_MASTER_SPEC.md` 为准；仍不明确时，停止扩展功能，优先选择更简单、更少信息的方案。
