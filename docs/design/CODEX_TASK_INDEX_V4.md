1. `ZEN_CANVAS_UIUX_BRAND_SYSTEM_V4.md`

1. 业务安全边界、数据安全和不可破坏行为优先；

3. 当前阶段任务书优先于旧页面的历史结构；
4. 不得用删除、弱化或绕过测试解决冲突。

## 3. 每阶段统一执行规则

### 开始前

```bash
git status --short
npm run typecheck
npm test
```

若基线失败：

- 停止大范围修改；
- 完整汇报失败命令和错误；
- 区分环境问题、历史问题和本阶段问题；
- 等待确认。

### 实施中

- 不安装新依赖，除非任务书明确允许并得到确认；
- 不修改任务范围外的 Store、Tauri、Rust、数据库和业务逻辑；
- 不删除、放宽或绕过已有测试；
- 不把调试页、视觉测试页或模拟数据带入生产路由；
- 不使用浏览器原生 `confirm`；
- 不新增硬编码 Tailwind 调色板或十六进制颜色；
- 不在普通用户界面展示 Batch Size、并发、Token、JSON 等工程参数；
- 所有新增组件同时考虑亮色、深色、Windows、macOS 和键盘操作。

### 完成后

```bash
npm run typecheck
npm test
npm run build
git diff --check
git status --short
```

每阶段独立提交，不要 amend 已验收提交。

### 汇报格式

1. 修改文件及目的；
2. 核心设计与交互变化；
3. 保留的兼容性和安全边界；
4. 新增/修改测试；
5. 三条命令的完整结果；
6. 未完成事项和已知风险；
7. 指定尺寸的亮色、深色和交互状态截图；
8. 新提交 SHA；
9. 停止并等待验收。

## 4. 设计硬规则

- 主导航固定为：概览、文件库、整理建议、历史记录；
- 高级区固定为：自动化、设置；
- 每个页面只有一个 Primary；
- 列表优先于卡片网格；
- 卡片不能套卡片；
- 内容区不滥用毛玻璃；
- 技术参数渐进式披露；
- 任何移动、重命名、清理操作先预览；
- 清理默认进入系统回收站，不做永久删除；
- 操作完成必须提供清楚结果、失败项和恢复路径；
- 选中态、风险和成功不能只依赖颜色；
- Windows 不伪装 macOS，macOS 不伪装 Windows；
- 中文正文不低于 13px，关键交互控件边界和焦点满足无障碍对比度；
- 不用大段说明弥补不清楚的交互。

---

## 5. 终止条件

任何阶段出现以下情况，应停止并汇报，而不是自行扩大范围：

- 需要改变数据库结构；
- 需要新增 Rust/Tauri 能力；
- 现有后端数据不足以完成设计；
- 业务安全规则与设计要求冲突；
- 需要新增第三方依赖；
- 需要修改尚未进入本阶段的核心页面；
- 测试暴露出历史业务缺陷而非当前 UI 回归。
